# エピソード: Claude Codeの/batchと/simplify——実装後の「面倒な仕上げ」を自動化する

## 台本

MC: 今回のテーマは「Claude Codeの/batchと/simplify」です。2月末にリリースされたv2.1.63で追加された2つのスラッシュコマンドなんですが、これがかなり実用的だと話題になっています。

ゲスト: よろしくお願います。この2つ、名前だけ見ると地味なんですけど、エンジニアの日常ワークフローで一番時間を食う「実装後の仕上げ」を自動化してくれるんですよね。コードを書くこと自体はClaude Codeが得意ですけど、書いた後のレビュー準備やリファクタリング、大規模な移行作業って、今まで人間がやるしかなかった。そこを埋める機能です。

MC: まず/simplifyから聞いていいですか？どんなコマンドなんでしょう。

ゲスト: 一言でいうと「実装後のセルフレビューを自動化するコマンド」です。セッション内で/simplifyと打つだけで、直前の変更に対して3つの並列エージェントが同時にコードをチェックして、問題があればその場で修正してくれます。

MC: 3つの並列エージェントというのは？

ゲスト: それぞれ観点が違うんです。1つ目が「重複検出エージェント」。同じようなロジックが複数箇所に書かれていないか、共通化できるパターンがないかを探します。2つ目が「品質チェックエージェント」。ネストが深すぎる三項演算子をswitch文に変えるとか、コードの可読性やメンテナンス性を改善する提案をします。3つ目が「効率エージェント」。不要な処理や無駄な計算がないかを検出します。この3つが並列で動くので、順番にチェックするより速い。

MC: 機能は変えずに、コードの構造だけ改善するってことですか。

ゲスト: まさにそうです。ここが重要なポイントで、/simplifyは既存のテストがすべて通る状態を維持したまま、コードの内部構造だけを改善します。動作は変わらない。だから安心して使える。

MC: 具体的にはどういうタイミングで使うのが効果的ですか？

ゲスト: 一番おすすめのワークフローは、Claude Codeに機能を実装させた後、一度git addしてコミットして、そのタイミングで/simplifyを実行する。すると変更されたファイルのdiffを見て、3つのエージェントがチェックして修正を適用してくれます。その後、git diffで修正内容を確認して、OKならもう一度コミット。実装コミットとクリーンアップコミットが分かれるので、レビューする側もわかりやすいんです。

MC: PRの品質が上がりそうですね。時間的にはどれくらい変わりますか？

ゲスト: Qiitaに実践記事を書いた方の計測だと、従来の手動レビューとクリーンアップで約30分かかっていたのが、/simplifyの実行と確認で約7分。PRあたり23分の短縮ですね。1日に3本PRを出す人なら、1時間以上浮く計算です。

MC: それは大きいですね。オプションもあるんですか？

ゲスト: はい。/simplifyだけで全体チェックもできますし、「/simplify focus on security patterns in the auth flow」のように観点を絞ることもできます。あと地味に強力なのが、CLAUDE.mdとの連携です。チームのコーディング規約をCLAUDE.mdに書いておけば、/simplifyがその規約に沿っているかも自動でチェックしてくれる。人間のレビュアーが「うちのルールだとこう書いて」って毎回指摘してたのを、自動化できるわけです。

MC: では次に/batchについて。こちらはどういうコマンドですか？

ゲスト: /batchは「大規模な一括変更を並列エージェントで自動実行する」コマンドです。たとえば「プロジェクト全体でaxiosをfetchに置き換えて」とか「全ファイルにTypeScriptの型注釈を追加して」みたいな、数十ファイルにまたがる変更を自然言語で指示するだけで、自動的に分割・並列実行・PR作成までやってくれます。

MC: 数十ファイルの変更を1つのコマンドで？

ゲスト: はい。内部的には3つのフェーズで動きます。第1フェーズが「調査と計画」。オーケストレーターエージェントがコードベースを調査して、作業を5から30の独立したユニットに分割します。ここがポイントで、各ユニットは他のユニットに依存しない、独立してマージできる単位になっている。分割計画はユーザーに提示されるので、実行前に確認できます。

MC: 計画を見てから「これでいいよ」って承認するわけですね。

ゲスト: そうです。第2フェーズが「ワーカーの並列実行」。承認すると、各ユニットごとにバックグラウンドエージェントが起動します。ここで面白いのが、各エージェントがGit Worktreeという仕組みで完全に分離された作業環境を持つこと。.claude/worktrees/ディレクトリの中に、エージェントごとに専用のブランチと作業コピーが作られるんです。だからエージェント同士がファイルを上書きし合ったり、マージコンフリクトを起こしたりしない。

MC: 各エージェントが自分だけの作業部屋を持っている感じですね。

ゲスト: まさにそのイメージです。しかも各ワーカーは、実装が終わったら自動でテストを実行して、さらに先ほどの/simplifyでコードレビューもかけた上でPRを作成する。つまり第3フェーズの「PR作成」の時点で、各PRはテスト済み・レビュー済みの状態で出てくるんです。

MC: /simplifyが内蔵されているのは賢い設計ですね。どういう場面で使うのが向いてますか？

ゲスト: ファイル間の依存関係が弱い一括変更に向いています。具体例だと、フレームワークの移行。ReactからVueとか。ライブラリの置き換え。axiosをfetchにとか。命名規則の統一、キャメルケースをスネークケースに変えるとか。API契約が変わって呼び出し側を全部直す場合なんかも得意です。逆に向かないのは、あるエージェントの出力が別のエージェントの入力になるような依存関係の強い変更や、ゴールが明確でない探索的なリファクタリングです。

MC: 時間的な効果は？

ゲスト: Qiitaの記事で報告されていた例だと、50ファイルにまたがる移行作業が、手動だと丸1日かかるところ、/batchで約30分。しかも各PRが独立しているので、1つのユニットが失敗しても他に影響しない。全部を1つの巨大PRにまとめるリスクがなくなるんです。

MC: 安全性と速度が両立するわけですね。ちなみに、/batchとヘッドレスモードの使い分けはどうですか？

ゲスト: いい質問です。/batchはインタラクティブなセッション内で使う一括変更コマンドで、計画の承認や進捗の確認ができます。一方、ヘッドレスモードはclaude -pで起動する非対話型モードで、CI/CDパイプラインに組み込むのに適しています。たとえばGitHub ActionsでPRが作られるたびにセキュリティ分析を走らせるとか、GitLab CIでコード品質チェックを自動化するとか。ユースケースが違うんです。プロジェクト全体の移行作業なら/batch、CI/CDの自動化ならヘッドレスモード、という使い分けですね。

MC: エンジニアが明日から使うとしたら、まず何から始めるのがいいですか？

ゲスト: 3ステップをおすすめします。まず、普段の開発で実装後に/simplifyを打つ習慣をつけること。これだけでPRの品質が上がるのを実感できるはずです。次に、CLAUDE.mdにチームのコーディング規約を書いておくこと。/simplifyがそのルールを参照してチェックしてくれるようになるので、チーム全体のコード品質が底上げされます。そして3つ目、溜まっていた移行作業やリファクタリングがあれば/batchを試してみる。「全ファイルのdeprecatedなAPIを新しいAPIに置き換えて」みたいな、今まで「やりたいけど面倒で手をつけてなかった」作業が一番効果を実感しやすいです。

MC: ありがとうございます。まとめると、/simplifyは「実装後の品質チェックとクリーンアップの自動化」、/batchは「大規模な一括変更の並列実行とPR自動作成」。どちらもエンジニアのワークフローで時間を食っていた部分を的確に自動化してくれる。特にこの2つが連携していて、/batchの中で/simplifyが自動実行される設計は、品質を落とさずにスピードを上げるという理想を実現してますね。今回は「Claude Codeの/batchと/simplify、実装後の面倒な仕上げを自動化する」をお届けしました。
