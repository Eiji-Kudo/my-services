# エピソード: Claude Codeの/batchと/simplify——実装後の「面倒な仕上げ」を自動化する

- ソース: Claude Code v2.1.63リリース情報、SmartScope技術ガイド、Qiita実践記事、エンジニアブログ等を統合（2026年3月）

## 台本

MC: PRを出す前のコード整理って、気が重くないですか。機能は動いてる。でも「あの関数どこかで書いたな」って探したり、変数名が雑だなって直したりして、気づいたら30分。今回紹介するClaude Codeの新コマンドは、まさにそこを自動化してくれます。テーマは「/batchと/simplify」です。

ゲスト: よろしくお願いします。コードを書いた"後"の作業って、意外と時間かかりますよね。そこを自動化する2つのコマンドが、2月末のv2.1.63で追加されました。

MC: まず/simplifyから聞いていいですか？

ゲスト: 一言でいうと「実装後のセルフレビューを自動化するコマンド」です。Claude Codeとの会話の中で/simplifyと打つだけ。裏側で3つのAIが並列に動いて、それぞれ別の観点でコードをチェックして、問題があればその場で修正してくれます。

MC: 3つに分けるのはなぜですか？

ゲスト: コードレビューで一番よくある指摘って「これ、同じ処理が別の場所にもありますよね」っていう重複の指摘じゃないですか。1つ目のAIはまさにそれを探す専門家です。同じロジックが複数箇所に散らばっていないか、共通化できないかをチェックする。残りの2つはコードの読みやすさと無駄な処理を担当しています。3つが同時に動くから速い。

MC: 機能は変えずに、コードの構造だけ改善するってことですか。

ゲスト: そうです。既存のテストがすべて通る状態を維持したまま、コードの内部構造だけを改善します。動作は変わらない。だから安心して使える。

MC: どういうタイミングで使うのが効果的ですか？

ゲスト: 一番おすすめは、Claude Codeに機能を実装させた後、一度変更をコミット——保存ですね——して、そのタイミングで/simplifyを走らせるワークフローです。3つのAIがチェックして修正を適用してくれる。その後、修正内容を確認してOKならもう一度コミット。実装のコミットとクリーンアップのコミットが分かれるので、レビューする側も差分が追いやすいんです。

MC: 時間的にはどれくらい変わりますか？

ゲスト: ある開発チームの計測では、1日3本PRを出す人で1時間以上浮く計算になっています。1PRあたり、30分かかっていた手動のレビューとクリーンアップが約7分になるので。

MC: それは大きいですね。チームの規約との連携もできるんですか？

ゲスト: はい。CLAUDE.mdという、Claude Codeにプロジェクトのルールを伝える設定ファイルがあるんですが、ここにチームのコーディング規約を書いておくと、/simplifyがその規約に沿っているかも自動でチェックしてくれる。レビュアーが「うちのルールだとこう書いて」って毎回指摘してたのを、自動化できるわけです。

MC: なるほど。CLAUDE.mdで規約チェックまで自動化できるなら、ファイルが何十個もあるような大規模な変更にも対応できるんですか？

ゲスト: そこは/batchの出番です。/batchは「大規模な一括変更を並列で自動実行する」コマンドで、数十ファイルにまたがる変更を自然言語で指示するだけで、分割・並列実行・PR作成までやってくれます。

MC: 具体的にはどう動くんですか？

ゲスト: たとえば50個のファイルで使っている古いライブラリを、新しい書き方に全部置き換えたいとします。/batchを実行すると、まず全体を指揮するAIがコードベースを調査して、「この50ファイルを10グループに分けよう」と計画を立てます。各グループは他のグループに依存しない、独立してマージできる単位です。この計画がユーザーに提示されるので、「これでいいよ」と承認してから実行に移ります。

MC: 計画を確認してからスタートするんですね。その後は？

ゲスト: 承認すると、各グループごとにAIが起動して並列で作業を始めます。グループ分けの段階で担当ファイルが重複しないように振り分けているので、各AIが完全に独立した環境で作業できるんです。

MC: つまりAI同士が並行作業しても、お互いの変更がぶつかったりしない？

ゲスト: ぶつからないです。しかも各AIは、実装が終わったら自動でテストを実行して、さらに先ほどの/simplifyでコードレビューもかけた上でPRを作成する。つまりPRが出てくる時点で、テスト済み・レビュー済みの状態なんです。

MC: /simplifyが自動で組み込まれている。時間的にはどれくらい違いますか？

ゲスト: SmartScopeの検証だと、50ファイル規模の移行作業が手動で丸1日かかるところ、/batchで約30分。しかも各PRが独立しているので、1つが失敗しても他に影響しない。全部を1つの巨大PRにまとめるリスクがなくなるんです。

MC: 安全性と速度が両立するわけですね。逆に向かない場面はありますか？

ゲスト: あるAIの作業結果が別のAIの入力になるような、依存関係の強い変更には向きません。あくまで「同じパターンの修正を多数のファイルに適用する」タイプの作業が得意です。

MC: エンジニアが明日から使うとしたら、何から始めるのがいいですか？

ゲスト: 今日から使えるのは/simplifyです。実装後にこれだけ打つ習慣をつけると、PRの品質が上がるのをすぐ実感できるはずです。/batchは、今まで面倒で後回しにしてた大きめの移行作業やリファクタリング、それを試す機会に使ってみてください。「古いAPIを新しいAPIに全ファイルで置き換えて」みたいな、やりたいけど手をつけてなかった作業が一番効果を感じやすいです。

MC: コードを書いた後の面倒な仕上げが、今日から自動化できる。ぜひ試してみてください。今回は「Claude Codeの/batchと/simplify」をお届けしました。
