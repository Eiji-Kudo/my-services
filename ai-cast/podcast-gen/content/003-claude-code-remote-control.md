## 台本

MC: 今回のテーマは「Claude Code Remote Control」。2月25日にAnthropicがリリースした新機能で、Zennでも120いいね超えの記事が出るなど注目を集めています。「ローカルのClaude Codeセッションをスマホから操作できる」って、一体どういうことなんですか？

ゲスト: よろしくお願いします。まさに名前の通りで、PCのターミナルで動いているClaude Codeのセッションを、スマホやタブレット、別のブラウザからそのまま操作できるようになったんです。ポイントは「ローカルで動き続ける」こと。コードもファイルもMCPサーバーも全部自分のマシン上にあるまま、操作だけをリモートからできる。

MC: 前回のエピソードでOpenClawの話をしたとき、「スマホだけで開発が完結する」というのが衝撃的でしたけど、それに近い体験がClaude Codeでもできるようになったわけですね。

ゲスト: そうなんです。ただ、OpenClawとは根本的にアプローチが違います。OpenClawは常駐型で、VPSやMac miniで24時間動いて、SlackやDiscord経由でやり取りする。一方、Remote ControlはあくまでローカルのClaude Codeセッションへの「窓」を開けるだけ。新しいサーバーを立てる必要もなければ、常駐デーモンも不要。今ある開発環境をそのまま使えるのが強みです。

MC: 使い方はどれくらい簡単なんですか？

ゲスト: びっくりするほど簡単です。2つの方法があります。1つ目は新しいセッションを始める場合で、ターミナルで「claude remote-control」と打つだけ。するとQRコードとURLが表示されるので、スマホのClaudeアプリでスキャンすれば接続完了。2つ目は既に作業中のセッションをリモートに切り替える場合で、セッション内で「/rc」と打つだけです。こちらは会話履歴をそのまま引き継げるので、デスクで途中まで進めた作業をソファに移動してスマホから続ける、みたいな使い方ができます。

MC: /configで全セッション自動有効化もできるんですよね。

ゲスト: はい。「すべてのセッションでリモートコントロールを有効にする」をtrueにすると、毎回コマンドを打たなくても常にリモート接続可能な状態になります。あと地味に便利なのが、事前に「/rename」でセッションに名前をつけておくと、スマホ側のセッション一覧で見つけやすくなること。「フロントエンドリファクタ」とか「API設計」とか名前をつけておくと、複数セッション走らせてるときに迷わない。

MC: さて、ここで気になるのがセキュリティです。ローカルのマシンにリモートからアクセスできるとなると、外部からの穴が開くんじゃないかと心配する人も多いと思います。

ゲスト: まさにその点を検証したZennの記事があって、aromariousさんがlsofコマンドで実際の通信を観察してるんです。結論から言うと、Remote Controlを有効にしても、ローカルマシンに新しいリスニングポートは一切開きません。通信はすべてアウトバウンドのHTTPS、つまり自分のマシンからAnthropicのAPIサーバーへの外向き通信だけ。

MC: ngrokみたいにインバウンドの穴が開くわけじゃないんですね。

ゲスト: そこが決定的に違います。ngrokは汎用のトンネルで任意のTCPやHTTPトラフィックを通せますが、Remote Controlはアプリケーション限定のリレーです。流れるのはClaude Codeのチャットメッセージとツール実行結果だけ。ファイルの中身やMCPサーバーの情報がそのまま外に出るわけではない。認証もClaudeアカウントのセッション管理に紐づいていて、ngrokのような公開URLは発行されません。

MC: 通信の暗号化はどうなってますか？

ゲスト: すべてのトラフィックがTLS経由でAnthropicのAPIを通過します。加えて、複数の短命なクレデンシャルを使っていて、それぞれ単一の目的にスコープされて独立して期限切れになる。ただし、aromariousさんが正直に書いているのが、TLSの実装詳細やセッショントークンの生成ロジック、アカウントとデバイスの紐づけの堅牢性は外部からは検証できないという点。「安全かどうか」ではなく「何が事実で、何が未検証で、信頼境界がどこにあるか」を理解することが大事だと。

MC: なるほど、技術的に穴は開いていないけど、Anthropicのサーバーを信頼するという前提は通常のClaude Code利用と同じだと。

ゲスト: その通りです。普段Claude Codeを使ってる時点でAPIサーバーとの通信は発生してるわけで、Remote Controlが追加するリスクは「外部からアクセスする経路が増えた」ことだけ。セッションURLを知っている人はそのセッションを完全にコントロールできるので、URLはクレデンシャルと同じように扱う必要があります。

MC: 実際に使ってみた人のフィードバックはどうですか？

ゲスト: Simon Willison氏がリリース当日に試していて、モバイルアプリのCodeタブに「Remote Control Session (Mac)」と表示されてシームレスに操作できたと。ただ、現時点ではdangerously-skip-permissionsフラグが使えないので、ファイル編集やコマンド実行のたびにスマホ側で承認が必要になる。長時間の自律的なタスクをスマホから放置するという使い方はまだ難しい。また、セッションが500エラーに落ちるケースもあったようで、リサーチプレビューらしい不安定さは残ってます。

MC: 現状の制限を整理すると？

ゲスト: 3つあります。1つ目、Maxプランが必要。Proプランへの展開は近日中とのことですが、まだです。2つ目、ターミナルを閉じるとセッションが切れる。PCがスリープから復帰すれば自動再接続しますが、ターミナル自体を閉じたらアウト。3つ目、約10分以上ネットワークに繋がらないとタイムアウトで終了する。なので「PCをシャットダウンして外出先からスマホで作業」は今のところできません。

MC: 最後に、エンジニアとしてRemote Controlをどう活用するのがいいか、おすすめの使い方を教えてください。

ゲスト: 3つのシナリオが特に合ってます。1つ目は「長時間タスクの監視」。テストスイートの実行やリファクタリングなど、時間のかかるタスクを走らせてデスクを離れ、スマホで進捗を見て追加指示を出す。2つ目は「レビューと軽い修正」。電車の中でPRのコードを読んでもらって、簡単な修正はそのまま指示する。3つ目は「アイデアの即実行」。散歩中に思いついた実装方針を、ローカル環境の文脈を持ったClaude Codeにすぐ伝えて着手させる。いずれも、ローカルのファイルシステムやMCPサーバーがそのまま使えるからこそ成り立つユースケースです。

MC: ありがとうございます。Claude Code Remote Controlは「ローカル環境の強みを保ったまま、場所の制約から解放される」という、シンプルだけどインパクトの大きい機能ですね。リサーチプレビューなので荒削りな部分はありますが、開発スタイルが変わるポテンシャルは十分。今回は「Claude Code Remote Controlで変わる開発スタイル」をお届けしました。
